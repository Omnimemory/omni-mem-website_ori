# 系统子模块架构文件

更新时间：2026-01-10  
范围：基于 Qbrain_Saas 代码整理子模块职责、接口、输入/输出与依赖。

## 1. 子模块清单与职责

1) gateway（apps/gateway）
- 角色：对外统一入口，鉴权/签发内部 JWT，路由到业务 API / Supabase Auth / Supabase REST。
- 关键能力：API Key 校验、用户 Token 校验、内部 JWT（x-api-token）签发、请求透传与白名单。

2) api（apps/api）
- 角色：业务 API 层，封装账户、密钥、LLM 配置、上传、用量、记忆服务入口。
- 关键能力：内部 JWT 鉴权、限流/配额预扣、队列写入、Supabase 数据管理、对象存储上传。

3) worker（apps/worker）
- 角色：异步任务执行与配额/用量同步。
- 关键能力：上传处理（从 OSS 拉取→解析→写入 core）、记忆任务（ingest/retrieval/status/graph）、配额回写。

4) Supabase（postgrest + gotrue）
- 角色：账户/配置/密钥等结构化数据存储 + 用户认证。
- 关键能力：API Key 表、LLM Key 表、账户权益、上传状态、用量账本。

5) Redis + BullMQ
- 角色：任务队列与配额缓存。
- 关键能力：{QUEUE_PREFIX}_upload、{QUEUE_PREFIX}_memory 两条队列；配额预扣、并发冲突控制。

6) OSS/MinIO
- 角色：上传文件存储。
- 关键能力：预签名上传 / 直传；回调触发入队。

7) Core Memory Service（mem/mock_core）
- 角色：记忆核心能力（向量/图谱/检索）。
- 关键能力：对话 ingest、retrieval、任务状态与图谱查询。

## 2. 主链路与调用关系（简化）

- 外部请求 → gateway → api → (Supabase / Redis / OSS / worker) → core
- worker → core 执行长任务 → 结果回写 Supabase / Redis
- gateway 负责把外部 API Key / 用户 Token 换成内部 JWT（x-api-token）

## 3. 子模块内部逻辑与接口（输入/输出）

### 3.1 gateway

输入：
- HTTP 请求：/api/v1/*、/auth/v1/*、/rest/v1/*、/health
- 认证头：Authorization: Bearer <user_token | api_key> 或 X-Api-Key
- 可选：X-Request-Id、X-Debug-Core

处理：
- 读取 Supabase REST：api_keys、accounts（校验 API Key、读取 scope）
- 读取 Supabase Auth：/auth/v1/user（校验用户 Token）
- 生成内部 JWT（x-api-token）并注入请求头：
  - x-api-token（内部 JWT）
  - x-tenant-id（账户/租户）
  - x-principal-user-id
  - x-principal-apikey-id

输出：
- 转发到 api：/api/v1/*（大多数业务请求）
- 转发到 Supabase Auth：/auth/v1/*
- 转发到 Supabase REST：/rest/v1/*

边界与策略：
- PUBLIC_WHITELIST 内路径可跳过 API Key 校验（如密码重置）
- X-Debug-Core 仅允许 admin scope

### 3.2 api

输入：
- HTTP 请求（需 x-api-token 内部 JWT）
- 账号上下文：x-tenant-id / x-principal-*（由 gateway 注入）

核心依赖：
- Supabase REST：accounts、api_keys、llm_keys、memory_policy、uploads、usage_ledger 等
- Redis：配额、幂等、并发控制
- BullMQ：上传队列、记忆队列
- OSS/MinIO：文件直传、预签名
- Core Service：记忆检索/写入

输出：
- 对外管理 API（API Key、LLM Key、Settings、Usage）
- 记忆 API（ingest/retrieval/status/graph）
- 上传 API（init/direct/history/status）

并发保护（新增）：
- core 请求超时封顶：`CORE_REQUEST_TIMEOUT_MS`（默认 8000ms），用于 retrieval 与 job/session 请求。
- job/session 状态缓存：`CORE_STATUS_CACHE_TTL_SECONDS`（默认 2s）+ `CORE_STATUS_CACHE_KEY_PREFIX`，降低轮询压力。

### 3.3 worker

输入：
- 上传队列 {QUEUE_PREFIX}_upload
- 记忆队列 {QUEUE_PREFIX}_memory

处理：
- 上传任务：从 OSS 拉取 → 解析 JSON/JSONL → 调用 core /ingest/dialog/v1 → 更新 uploads 状态 → 记账（usage_ledger）
- 记忆任务：按 action 调用 core /retrieval/dialog/v2、/ingest/jobs/*、/graph/* 等
- OpenRouter 管理：记录用量、更新 key 状态
- 配额同步：Redis → Supabase accounts.balance
- Qdrant 节点统计：ingest 成功后按 tenant_id 或 user_id（`u:<account_id>`）统计多 collection 节点数，汇总写入 accounts.public_data.qdrant_node_counts。
  - Note：曾出现节点数始终为 0 的根因是过滤字段与写入 metadata 不一致（只按 user_id 纯值统计，实际 payload 为 tenant_id 与 `u:<id>`）。现已兼容两种字段。

输出：
- 更新 Supabase 数据（uploads、usage_ledger、openrouter_usage_events、accounts）
- 更新 Redis 配额/状态

## 4. 对外 SDK 接口清单（经 gateway 访问）

说明：
- Base URL：{GATEWAY}/api/v1
- 鉴权：
  - API Key：Authorization: Bearer qbk_* 或 X-Api-Key: qbk_*
  - 用户 Token（Supabase access_token）：Authorization: Bearer <jwt>
- 幂等：建议传 X-Request-Id（服务端用于去重/重试）

### 4.1 记忆类接口（SDK 推荐）

1) POST /api/v1/memory/ingest
- 参数：	urns[]（数组）、可选 session_id、client_meta
- 行为：入队 → worker → core /ingest/dialog/v1
- 结果：返回 core 回执（job_id/status）

2) POST /api/v1/memory/retrieval
- 参数：query（必填）、可选 client_meta
- 行为：直接调用 core /retrieval/dialog/v2
- 结果：返回检索内容

3) GET /api/v1/memory/ingest/jobs/:job_id
- 参数：job_id（路径）
- 行为：查询 core ingest job 状态

4) GET /api/v1/memory/ingest/sessions/:session_id
- 参数：session_id（路径）
- 行为：查询 core ingest session 状态

5) POST /api/v1/memory/graph/v1/search
- 参数：请求体（graph 查询）
- 行为：透传到 core /graph/v1/search

6) GET /api/v1/memory/graph/v0/*
- 参数：graph path（路径）
- 行为：透传到 core /graph/v0/*

### 4.2 上传与批量记忆（SDK 可选）

1) POST /api/v1/uploads/init
- 参数：filename、size、mime、可选 api_key_id、force
- 结果：返回 upload_url（OSS 预签名）与 upload_id

2) POST /api/v1/uploads/direct（multipart/form-data）
- 参数：文件流 + filename / size / mime / 
equest_id
- 行为：直传 OSS → 入队处理

3) GET /api/v1/uploads/:id
- 参数：id
- 结果：上传任务状态（init/uploaded/processing/done/failed）

4) GET /api/v1/uploads/history
- 参数：page、pageSize
- 结果：上传历史与用量

### 4.3 管理类接口（控制台/管理 SDK）

- API Key：GET /apikeys、POST /apikeys、POST /apikeys/:id/reveal、POST /apikeys/:id/rotate（legacy；api 返回 403 `rotate_disabled`，前端仍保留入口）、PATCH /apikeys/:id、DELETE /apikeys/:id
- 账号设置：PATCH /settings/account-name
- 记忆策略：GET /settings/memory-policy、PUT /settings/memory-policy
- 套餐权益：GET /settings/scope、POST /settings/scope-upgrade
- LLM Keys：GET /llm-keys、POST /llm-keys、PUT /llm-keys/:id、DELETE /llm-keys/:id、POST /llm-keys/:id/disable、POST /llm-keys/:id/enable
- 模型列表：GET /llm-models?provider=...（需 X-LLM-Api-Key）
- 用量与余额：GET /usage/summary、GET /usage/ledger、GET /balance
- IAM：POST /iam/authorize（仅 Bearer token）

### 4.4 公开（无需 API Key）接口

- POST /auth/password-reset/request
- POST /auth/password-reset/verify
- POST /auth/password-reset/confirm
- GET /health

## 5. 关键约束与注意事项

- 内部鉴权：api 只接受 gateway 生成的 x-api-token（内部 JWT）
- 幂等与重试：创建/轮换 API Key、上传初始化依赖 X-Request-Id
- 配额：上传按大小预扣，失败回补；usage_ledger 记录用量
- Debug：X-Debug-Core 仅 admin scope
