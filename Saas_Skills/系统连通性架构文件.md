## 6. 组件任务边界（SaaS 主链路）

- `kong`：公网入口路由，不做业务补全。  
- `qbrain-gateway`：鉴权、签发 internal JWT、将外部请求转发到 `api-dev`；读取 Supabase（postgrest/gotrue）。  
- `api-dev`：业务补全、权限/配额检查、同步调用 `mem`；写侧把耗时任务入 Redis 队列。  
- `api-dev`（并发保护）：对 core 请求设置超时上限；对 ingest job/session 状态查询做短 TTL 缓存（降低轮询压力）。  
- `worker-dev`：消费 Redis 队列，调用 `mem` 执行重任务（依赖业务实现幂等/重试）。  
- `mem`：对话写入、检索、搜索；依赖 `qdrant` 向量库、`neo4j` 图谱；使用外部 LLM Provider（OpenRouter/DashScope）。  
- `postgrest`/`gotrue`：提供配置、账号、密钥管理（Postgres 为底层存储）。  
- 观测：`otel-collector` 收集 Trace，`jaeger-all-in-one` 提供 UI。

## 7. 调试建议（core-debug 抓包）

`api-dev` 与 `worker-dev` 已开启 `CORE_DEBUG_LOG=true`。如需抓取请求/响应明细，在请求头加入 `x-debug-core: 1`（建议同时带 `X-Request-Id` 便于日志关联），再在对应 Pod 日志中过滤 `[core-debug]`。

## 8. core 超时与状态缓存（并发优化）

- 超时封顶：`CORE_REQUEST_TIMEOUT_MS`（默认 8000ms），覆盖 retrieval 与 job/session 的 core 请求。  
- 状态缓存：`CORE_STATUS_CACHE_TTL_SECONDS`（默认 2s）与 `CORE_STATUS_CACHE_KEY_PREFIX`，仅用于 job/session 状态查询。
