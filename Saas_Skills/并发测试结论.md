## 结论 1：Retrieval 并发在 C 中明显改善，但仍缺 429

对比数据：  
- 实验 A（12:45）：total=1754, errors=1707, p95=8681ms, first_error=concurrency=10, status=502  
- 实验 B（18:19）：total=1501, errors=1427, p95=8428ms, first_error=concurrency=10, status=502  
- 实验 C（20:12）：total=5293, errors=0, p95=1178ms, first_error=-, rate_429=0  

意义：  
- A/B 在并发=10 就出现 5xx，阈值未上移。  
- C 出现“0 错误 + p95 明显下降”，说明缓存 + qdrant 扩容对检索链路有效。  
- C 仍未触发 429，导致用例判定为 failed（must_see_429）。  

可能原因：  
- qdrant/neo4j 单实例（或读能力不足）成为共享瓶颈，mem 扩容只放大了下游压力。  
- gateway/api → mem 的连接池/超时设置过小，在并发=10 即触发 upstream 失败。  
- 证据补充：qdrant CPU 峰值从 12:45 的 47% 上升到 18:19 的 56%，说明下游向量存储压力在增加但未达到充分饱和。  
- C 的改动项（非 ingest 缓存 + qdrant=3）有效降低了下游负载，且错误消失，说明瓶颈更偏下游容量/热点读。  

## 结论 2：Ingest 在 C 中触发 429，p95 明显下降

对比数据：  
- 实验 A（12:45）：total=1679, errors=0, p95=3593ms  
- 实验 B（18:19）：total=1687, errors=0, p95=3849ms  
- 实验 C（20:12）：total=7287, errors=0, p95=909ms, rate_429=7218  

意义：  
- A/B 无错误但 p95 超标，吞吐上不去。  
- C p95 降至 < 1s，且出现大量 429，说明入口限流已开始生效，系统以“可控拒绝”保护后端。  

可能原因：  
- ingest 仍受 mem → core 或 worker 入队处理路径影响，队列/存储吞吐不匹配。  
- qdrant/neo4j 或 Redis/worker 的下游耗时成为瓶颈，导致排队延迟上升。  
- C 的 429 说明已有速率控制（可能在入口或 worker 入队），但仅对 ingest 生效。  

## 结论 3：Blast-radius 在 C 中稳定，但仍缺 429 保护

对比数据：  
- 实验 A（12:45）：total=1730, errors=1594, p95=8504ms, first_error=concurrency=20, status=502  
- 实验 B（18:19）：total=2154, errors=2009, p95=8595ms, first_error=concurrency=10, status=502  
- 实验 C（20:12）：total=5198, errors=0, p95=1165ms, first_error=-, rate_429=0  

意义：  
- A/B 表现为 5xx 爆炸式失败。  
- C 出现 0 错误且 p95 明显改善，说明缓存与 qdrant 扩容提升了共享检索的稳定性。  
- 但仍未触发 429（rate-limit 预期），导致测试判定失败。  

可能原因：  
- retrieval 共享 mem/qdrant/neo4j 资源，缺少租户隔离与限流保护。  
- 入口层未触发 429（rate-limit），导致 5xx 扩散。

可能原因：  
- 状态读取主要依赖 api + mem 的轻量查询，未深度触发 qdrant/neo4j。  

## 结论 5：Dashboard 负载整体稳定，但仍有零星 5xx 抖动

对比数据：  
- 实验 A：dashboard 各用例通过，偶发 599 出现在并发 30（写/读）。  
- 实验 B：dashboard 各用例通过，偶发 599 出现在并发 30~40（写/读）。  
- 实验 C：dashboard 各用例通过，account-name 更新在并发 40 出现 1 次 502。  

意义：  
- dashboard 不是主瓶颈，但高并发下仍有 upstream 抖动。  
- 进一步说明入口层连接池/超时策略需优化。  

可能原因：  
- gateway/api 的连接池或超时设置不足，压力下出现偶发 upstream 失败。  

## 结论 6：缓存 + qdrant 扩容显著提升检索稳定性，但 retrieval/graph 仍缺 429 保护

对比数据：  
- RL-RETRIEVAL-RAMP-01：A/B errors>1400, p95≈8.4s → C errors=0, p95=1178ms  
- RL-BLAST-RADIUS-01：A/B errors>1500, p95≈8.5s → C errors=0, p95=1165ms  
- RL-INGEST-RAMP-01：A/B rate_429=0 → C rate_429=7218（p95=909ms）  

意义：  
- 说明非 ingest 缓存 + qdrant pod 扩容有效降低检索链路压力，系统从“5xx 崩溃”转为“稳定响应”。  
- 入口层仍未对 retrieval/graph 生效 429，导致用例判定失败，且无法形成“可控拒绝”。  

可能原因：  
- retrieval/graph 的限流规则尚未启用，或阈值过高导致测试未触发。  
- 缓存命中后实际下游负载下降，进一步降低触发 429 的可能性。  

## 并发瓶颈判断（结合系统架构）

1) qdrant / neo4j 下游吞吐不足  
   - A/B retrieval 失败阈值一致，mem 扩容无改善迹象，瓶颈更可能在下游存储。  
   - 证据补充：qdrant CPU 峰值 12:45 为 47%，18:19 为 56%，负载上升与错误未改善并存，提示瓶颈可能在 I/O/锁/请求排队而非纯 CPU。  
   - 实验 C 扩容 qdrant 后 5xx 消失，间接验证“下游容量”是主要瓶颈。  
2) gateway/api → mem 连接/超时配置偏紧  
   - A/B 5xx 在并发=10 即出现，且错误以 502 形态稳定出现。  
   - C 仍有零星 502（dashboard 写），说明连接/超时仍需优化。  
3) 缺少有效限流/熔断（429 覆盖不完整）  
   - A/B 未出现 429，表现为 5xx 失败。  
   - C 仅 ingest 出现 429，retrieval/graph 仍未出现，说明限流覆盖不完整。
